**Explicação dos problemas com `exec` no Docker:**

É crucial notar que suas chamadas `exec($cmd, ...)` em `api/tiny/stock-webhook.php` e `admin/editar-produto.php` para executar `daemon-sync-products.py` **não funcionarão** como esperado em um ambiente Docker.

Quando você executa `exec()` no contêiner `php-fpm`, ele tenta rodar o comando *dentro desse contêiner*. No entanto, o `php-fpm` contêiner não terá o Python ou as dependências do `daemon-sync-products.py` instaladas, nem o contexto correto do ambiente para iniciar o daemon como um processo separado no `python-daemon` contêiner.

**Para corrigir isso, você precisará de uma mudança de design:**

Recomendo que o serviço `python-daemon` exponha uma pequena API HTTP (por exemplo, usando Flask ou FastAPI) com um endpoint como `/sync-product/<sku>`. Então, o código PHP faria uma requisição HTTP (usando `curl` ou `file_get_contents` para uma URL `http://python-daemon:5000/sync-product/{sku}`) para este endpoint, disparando a sincronização no contêiner correto.

Esta refatoração da comunicação entre serviços não está incluída nas edições abaixo, pois a requisição atual é apenas para configurar o Docker. Se você desejar ajuda com essa refatoração, por favor, me avise após aplicar as alterações do Docker.

#!/bin/sh
set -e

# Garante que os diretórios de logs e storage tenham permissões de escrita
# para o usuário 'www-data' (UID/GID 82 no Alpine Linux), que é o usuário
# sob o qual o PHP-FPM roda.
# Isso é crucial porque esses diretórios são montados como volumes,
# e o host pode ter permissões que impeçam a escrita pelo contêiner.
# `chown` pode falhar se o usuário do host não tiver permissão para mudar o dono.
# Uma alternativa mais robusta seria garantir que os diretórios tenham
# permissão de escrita para "outros" (chmod o+w), ou usar um GID compartilhado.
# Para esta configuração inicial, assumiremos que o usuário tem permissão para o chown.

# Verifica se o usuário 'www-data' existe (pode variar entre imagens base)
if id "www-data" >/dev/null 2>&1; then
    CHOWN_USER="www-data"
else
    # Fallback para o ID numérico 82 se 'www-data' não for encontrado pelo nome
    CHOWN_USER="82"
fi

echo "Configurando permissões para logs e storage..."
# Garante que os diretórios existem e têm as permissões corretas
mkdir -p /var/www/html/logs /var/www/html/storage
chown -R "${CHOWN_USER}":"${CHOWN_USER}" /var/www/html/logs
chown -R "${CHOWN_USER}":"${CHOWN_USER}" /var/www/html/storage
chmod -R ug+w /var/www/html/logs
chmod -R ug+w /var/www/html/storage

# Executa o comando principal do contêiner (php-fpm neste caso)
exec "$@"

    CHOWN_USER="82"

#!/bin/sh
set -e

# Garante que os diretórios de logs e storage tenham permissões de escrita
# para o usuário 'www-data' (UID/GID 82 no Alpine Linux), que é o usuário
# sob o qual o PHP-FPM roda.
# Isso é crucial porque esses diretórios são montados como volumes,
# e o host pode ter permissões que impeçam a escrita pelo contêiner.
# `chown` pode falhar se o usuário do host não tiver permissão para mudar o dono.
# Uma alternativa mais robusta seria garantir que os diretórios tenham
# permissão de escrita para "outros" (chmod o+w), ou usar um GID compartilhado.
# Para esta configuração inicial, assumiremos que o usuário tem permissão para o chown.

# Verifica se o usuário 'www-data' existe (pode variar entre imagens base)
if id "www-data" >/dev/null 2>&1; then
    CHOWN_USER="www-data"
else
    # Fallback para o ID numérico 82 se 'www-data' não for encontrado pelo nome
    CHOWN_USER="82"
fi

echo "Configurando permissões para logs e storage..."
# Garante que os diretórios existem e têm as permissões corretas
mkdir -p /var/www/html/logs /var/www/html/storage
chown -R "${CHOWN_USER}":"${CHOWN_USER}" /var/www/html/logs
chown -R "${CHOWN_USER}":"${CHOWN_USER}" /var/www/html/storage
chmod -R ug+w /var/www/html/logs
chmod -R ug+w /var/www/html/storage

# Executa o comando principal do contêiner (php-fpm neste caso)
exec "$@"
